# Customer Brief

## Customer Name: 제주항공 (Jeju Air)
## Industry: 항공 (Aviation / Low-Cost Carrier)
## Date: 2026-03-30
## Source: requirements-doc (JejuAir-AI-Powered-NOTAM-Analysis-System-Requirements.pdf)

## Pain Points:
1. 항공고정통신망(AFTN)을 통해 하루 수천 건의 NOTAM 전문이 실시간 수신되지만, 담당자 3명이 24시간 교대 근무로 수동 모니터링해야 함
2. 수신되는 NOTAM이 비정형 텍스트와 약어로 구성되어 있어 해석이 어렵고 시간이 많이 소요됨
3. 근무자가 수동으로 해석 및 영향성을 판단한 후, 중요 정보를 J-FOS(제주항공 시스템)의 REF BOOK에 직접 수기 작성해야 함
4. 긴급 NOTAM 발생 시 분석 지연으로 인한 운항 차질 리스크가 존재함
5. 수천 건의 루틴 NOTAM 필터링에 소모되는 단순 반복 업무 시간이 과도함
6. 승무원 브리핑 자료(Company NOTAM, Dispatcher Comment 등) 수동 제작에 업무 부하가 큼
7. 고숙련 운항관리사의 역량이 단순 해석 업무에 소모되어 항로 우회 결정 등 고부가가치 의사결정에 집중하지 못함

## Requirements:
1. **LLM 기반 NOTAM 중요도 점수화**: 과거 REF BOOK에 등재된 3년치 NOTAM의 핵심 패턴(Q-CODE, 특정 키워드)을 RAG 또는 LLM으로 학습하여, 신규 수신 NOTAM이 REF BOOK 등재 대상인지 확률로 분류
2. **Q 코드 기반 기본 판단**: Q 코드를 기반으로 NOTAM의 기본 중요도를 판단 (예: QMRLC = runway closed)
3. **공간/운항 스케줄 기반 종합 분석**: 공간 정보, 운항 스케줄 정보를 기반으로 NOTAM의 중요도를 종합적으로 분석 (예: 활주로 4개 공항에서 1개 폐쇄 vs 활주로 1개 공항에서 1개 폐쇄는 중요도 상이)
4. **운항 영향편/항로 자동 매칭**: LLM이 NOTAM 본문에서 핵심 정보 추출(영향 반경, 위경도 좌표, 제한 고도, 유효시간 등) 후, 사내 시스템의 자사 운항편 항로(NAVBLUE 항로, JFOS화 필요)를 자동으로 교차 분석하여 영향받는 특정 운항편 리스트업
5. **중요 NOTAM 자동 필터링**: 수천 건의 NOTAM 중 운항에 실질적으로 영향을 미치는 중요 NOTAM을 자동 필터링
6. **항로 영향도 분석 대시보드**: 항로별 NOTAM 영향도를 시각적으로 파악할 수 있는 대시보드
7. **브리핑 문서 자동 생성**: 운항관리사용 결재 문서(분석 완료된 중요 NOTAM 요약문) 자동 생성
8. **승무원 브리핑 자료 자동 생성**: DISP COMMENT, COMPANY NOTAM, 기타 브리핑 서류 초안 자동 생성
9. **의사 결정 가이드 제공**: 항로 변경 등 권고 사항 제공 (예: 인천-방콕 노선 1번 대신 98번 루트 사용 권고)
10. **NOTAM-항로/스케줄 매칭 알고리즘**: NOTAM의 시공간적 영향 범위와 자사 운항 항로/스케줄의 교차를 자동 계산

## Additional Context:
- **런칭 목표 시점**: 2026 하반기
- **사용자 유형**: 운항관리사 (Flight Dispatcher) — 24시간 교대 근무, 3명 체제
- **기존 시스템**: J-FOS (제주항공 운항관리 시스템), REF BOOK (중요 NOTAM 기록 장부)
- **외부 데이터 소스**: AFTN (항공고정통신망)을 통한 실시간 NOTAM 수신
- **항로 데이터**: NAVBLUE 항로 데이터 (JFOS화 필요)
- **NOTAM 형식**: ICAO NOTAM 표준 형식 — Q 코드, 위경도 좌표, 영향 반경, 유효 시간, 제한 고도 등 포함
- **데이터 규모**: 하루 수천 건 수신, 과거 3년치 REF BOOK 데이터 학습 대상
- **AI 기술 요구**: LLM 기반 분석, RAG(Retrieval-Augmented Generation) 패턴 학습
- **프로토타입 범위**: AWS 기반 프로토타입 — Amazon Bedrock을 활용한 실제 AI 기능 구현 필요

## Data Structure:
- **NOTAM 전문**: 비정형 텍스트 (ICAO 표준 형식, Q 코드, 위경도, 유효 시간, 본문)
- **REF BOOK**: 과거 3년치 중요 NOTAM 등재 기록 (학습 데이터)
- **운항편 정보**: 자사 운항 스케줄 (출발/도착 공항, 시간, 항로)
- **항로 데이터**: NAVBLUE 기반 항로 좌표 (waypoint, 경유지, 고도)
- **공항 정보**: 공항 코드, 활주로 수, 시설 정보

## Existing System:
- **J-FOS**: 제주항공 자체 운항관리 시스템 — 현재 REF BOOK 수기 작성, 브리핑 자료 수동 생성
- **AFTN**: 항공고정통신망 — NOTAM 실시간 수신 채널
- **NAVBLUE**: 항로 데이터 제공 시스템 (별도 JFOS화 작업 필요)
- 별도의 참고 문서: "AI 활용 과제 제안서_통제센터팀_NOTAM.xlsx" (PDF 내 첨부 참조로 언급됨)

## Assumptions:
- A1. 프로토타입에서는 실제 AFTN 연동 대신 샘플 NOTAM 데이터를 사용하되, AI 분석 기능은 Amazon Bedrock을 통해 실제 동작하도록 구현
- A2. REF BOOK 과거 데이터는 프로토타입에서 시드 데이터로 제공 (실제 3년치 데이터 접근 불가 가정)
- A3. NAVBLUE 항로 데이터는 프로토타입에서 샘플 항로 데이터로 대체
- A4. J-FOS 연동은 프로토타입 범위 밖이며, 독립 웹 애플리케이션으로 구현
- A5. 중요도 점수화는 확률값(0~1)으로 표현하며, 임계치 기반 필터링 적용
- A6. 대시보드는 지도 기반 시각화를 포함 (항로 위의 NOTAM 영향 영역 표시)
